  "topic": "dynamic-programming",// from the fixed taxonomy below

  "starter_code": {"python3": "...", "java": "...", "c++": "...", "c": "..."},

Stdout comparison is **trimmed** (leading/trailing whitespace ignored).

## Probe labels (MEMORY FUEL)

Hidden test cases carry a `probe` label naming the *kind* of input they attack
with. When a submission fails a probed test, that label flows into the player's
memory episode — the Boss learns **which kind of case breaks the player**, not
just pass/fail. Visible tests never carry probes.

| Label | Attacks |
|---|---|
| `edge:empty` | empty input / empty side of a computation |
| `edge:single` | n=1 / single element / one-char input |
| `edge:duplicates` | repeated values that break uniqueness assumptions |
| `edge:all-equal` | every value identical |
| `edge:sorted` / `edge:reverse-sorted` | already-ordered input (degenerate cases) |
| `edge:negative` | negative values (breaks `max=0` inits, bit tricks) |
| `edge:zero` | zero values / zero targets / k=0 |
| `edge:max-bounds` | values or sizes at the stated limits |
| `edge:overflow` | results exceeding 32-bit range (mod arithmetic, 64-bit) |
| `edge:no-solution` | the -1 / NO / 0 case |
| `edge:cycle` | self-loops, cycles where a DAG/list was assumed |
| `edge:disconnected` | unreachable components |
| `edge:unbalanced` | skewed trees, mismatched lengths, narrow grids |
| `perf:large_n` | maximum n — punishes wrong complexity class |
| `adversarial:worst-case` | the classic trap for the intended algorithm |
